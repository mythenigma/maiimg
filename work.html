Knowledge Base 使用原则

写作基线
  • 这个 tab 只记录已经从当前项目代码、页面结构、接口调用里确认过的能力与边界。
  • 后续写 blog 时，优先引用这里的事实，不要凭行业想象补功能。
  • 如果某项能力没有在这里确认，默认不要写成“产品已支持”。
  • 不要写夸张结论、绝对化承诺或无法从当前项目里确认的效果描述；如果拿不准，先问用户再写。

中文 Blog 文案基线

表达约束
  • 不要用“中文界面截图”“操作流程图（SVG）”“一句话总结”这类像内部备注或说明书目录的标题。
  • 标题要围绕读者问题来写，例如“你会在页面里看到什么”“这篇文章想解决什么”“结论很简单”。
  • 截图说明不要只描述界面名称，要解释这张图对读者有什么意义，例如“生成结果：同一份分享同时给你链接和二维码”。
  • 首屏导语要先讲场景和痛点，不要一上来堆功能名词。
  • 文案允许有吸引力，但所有能力点都必须和当前真实产品能力一致。

产品定位（已确认）

真实能力
  • MaiIMG 当前是一个以静态前端为主的媒体分享站点，核心流程是：上传媒体 → 生成分享链接与二维码 → 设置访问控制 → 查看访问记录 → 登录后在 dashboard 管理。
  • 主站入口在 index.html 与 maiimg.html，两页都直接调用 https://fetch.maipdf.com 的接口。
  • 项目已经不只是“图片图床”，前端文案和上传校验都明确包含 images + short videos。

上传与分享能力（已确认）

代码已证实
  • 支持格式：PNG、JPG、JPEG、GIF、WebP、MP4、MOV、WebM、M4V。
  • 单次分享上限：最多 25 个文件。
  • 图片大小限制：50MB；视频大小限制：100MB；视频时长限制：120 秒。
  • 生成结果会同时给出 分享链接 和 二维码，两者来自同一份分享结果。
  • 结果区还会生成 Access Records code，可跳到 tracking.html 查看访问记录。

访问控制（已确认）

代码已证实
  • 前端明确要求填写 View Limit、Time Control、Description。
  • Time Control 在产品文案里表示“每次可查看秒数 / per-view duration”；虽然部分旧变量名仍叫 password，但写 blog 时不要把它写成普通密码功能。
  • 有效期支持预设：1 hour、3 hours、24 hours、5 days、custom days、unlimited。
  • 有效期支持 UTC 与本地时区显示，界面里有对应的时区下拉与本地时间展示。
  • 已发出的分享支持禁用/删除，当前语义上等价于让链接失效；在 dashboard 中删除会把视图限制置为 0。

查看页体验（已确认）

代码已证实
  • gallery/index.html 与 media/index.html 都是竖向 Swiper 查看器，不是传统网格式相册页。
  • 图片支持滑动浏览与缩放；页面有 Swipe to view more • Pinch to zoom images 的交互提示。
  • 视频支持原生播放控件，查看器会在切换页面时暂停其他视频，并带视频预览首帧预热逻辑。
  • 查看页有进度条、分页点、上一张/下一张按钮，以及图片全屏按钮；当前全屏按钮在视频页会隐藏。
  • 微信内打开时，会引导到 wechat.html 提示“请在浏览器中打开”，并配合 special.maiimg.com 做中转。

登录与 Dashboard（已确认）

代码已证实
  • 登录走 Firebase Auth，支持 Google 登录与邮箱登录。
  • 登录后可进入 dashboard.html 查看自己的 galleries 列表。
  • Dashboard 已有的真实动作包括：刷新列表、查看状态、打开分享链接、打开访问记录、删除 gallery、Restore、Replace Mode。
  • Replace Mode 的真实语义是：选一个目标 A 和一个来源 B，把 B 的内容和设置复制到 A，但 A 原来的链接保持不变。
  • Restore 现在不是“独立恢复旧版本系统”，而是通过 Replace Mode 先选中被删 gallery 作为目标 A，再挑一个 source B 来复用原链接。

Tracking / Access Records（已确认）

代码已证实
  • tracking.html 会根据 gallery id 与 access code 请求 /api/get-access-logs。
  • 页面会展示 gallery 信息、统计信息和访问日志表，不只是单一明细列表。
  • 当前代码里明确渲染的统计维度包括：Total Access、Unique IPs、Different Browsers、First Access。
  • Tracking 页也支持删除 gallery，需要输入形如 DELETE-{id} 的确认码。

Blog 写作时不要乱写的内容

边界提醒
  • 不要写“无限张图片/无限容量单链接分享”，当前代码明确上限是 25 个文件。
  • 不要写成“团队协同审批系统”“多人批注系统”“在线审片评论流”“品牌资产管理平台（DAM）”，这些都没有在当前代码里确认。
  • 不要写“端到端加密”“私有化部署”“开放 API”“自动 AI 打标/搜索/识图”，当前项目里没有对应实现证据。
  • 不要把 Time Control 描述成普通密码输入框；更准确的表达是“单次查看时长/时间控制”。
  • 如果文章主题只是图片分享，不要强行把视频能力塞进标题和核心卖点；但如果写产品总览，可以明确说明站点也支持 short videos。
  • “访问记录 / access records”可以写，“高级商业 BI 分析平台”不要写太满；当前确认到的是日志、统计和访问明细，不是完整分析套件。

5) Dashboard 登出与首页登录状态同步

待做

问题：Dashboard 检测登录只读 sessionStorage.maiimg_jwt；登出时只清除 sessionStorage 并跳转 maiimg.html，未调用 Firebase signOut。结果：① 跳转到 maiimg 后 Firebase 仍为已登录，onAuthStateChanged 会再次写入 JWT，用户马上又显示已登录；② 多标签时，另一标签的 index 与 dashboard 各自一份 sessionStorage，在 dashboard 登出不会让另一标签的 index 登出。

修复要点：

  • 在 dashboard.html 的 logout() 中：引入 Firebase Auth（与 index/maiimg 同配置），先调用 signOut(auth)，再清除 sessionStorage，最后跳转 maiimg.html。
  • 可选（多标签一致）：登出时向 localStorage 写一条“登出”标记或发 BroadcastChannel 消息；index.html / maiimg.html 监听 storage 或 message，收到后执行 signOut + 清除 sessionStorage + 刷新或更新 UI。

涉及文件：dashboard.html（登出逻辑 + 可选 Firebase 引入）、index.html / maiimg.html（可选：监听跨标签登出）。

6) 项目看法与改善建议（Roadmap）

建议

当前判断：MaiIMG 已具备“上传-分享-追踪-管理”的完整闭环，功能已经不弱；下一步重点应从“能用”转向“稳定转化 + 性能体验 + 内容增长”。

  • P0（优先）：登录转化与新功能可见性：未登录上传提醒继续保留；在生成成功区增加“登录后可管理历史链接/替换/访问记录”的固定提示；把“到期时间设置”放入首页与 blog 的首屏文案。
  • P0（优先）：移动端体验一致性：统一 index / maiimg / dashboard 的头部、按钮高度、卡片间距；减少手机版操作跳转层级，关键动作保持单屏可见。
  • P1：Gallery 加载策略：保持逐张懒加载，增加“缩略图 + 原图”两级资源与下一张预加载，优先提升弱网首屏速度，不建议先做复杂 chunk。
  • P1：数据与埋点：补齐关键漏斗事件（打开上传、开始上传、生成成功、点击登录、完成登录、进入 dashboard），每周看转化率再迭代文案和交互。
  • P1：SEO/知识库体系：围绕真实功能写主题（过期时间、访问记录、A/B 替换、登录管理）；每篇固定结构：问题场景 → 操作步骤 → 常见问题 → 相关功能入口。
  • P2：可信度建设：补“隐私与存储时长说明”“游客与登录用户能力对比”“常见错误码说明”，降低新用户不确定感。

建议执行方式：先做 P0 两周冲刺（转化+移动端），再做 P1（性能+内容+埋点），每周复盘一次核心指标。
